## 1. Set The Server Host

All API reference pages are configured to use:

```text
https://sandbox-api.cardcore.cloud
```

You can send requests directly against the Cardcore sandbox from the API playground.

## 2. Configure Authentication

The API uses three main auth patterns:

- `Bearer {{institutionToken}}` for most institution-scoped operations.
- `Bearer {{oauthToken}}` as an alternative for some user-scoped calls when scopes and permissions allow it.
- `x-api-key: {{apiKey}}` for `Import Card`.

The core runtime value you will need in most flows is:

- `institutionId`

The request examples also imply additional runtime values you will usually need during testing:

- `institutionToken`
- `oauthToken`
- `oauthRefreshToken`
- `cardId`
- `institutionUserId`
- `cardProgramId`
- `apiKey`

## 3. Start With Public Endpoints

These endpoints are the fastest way to validate your server configuration:

- `GET /util/banks`
- `POST /util/name-enquiry`

After those work, move to institution-authenticated endpoints such as:

- `GET /institutions/oauth/applications`
- `POST /institutions/cards`

## 4. Token Bootstrap

If you need an OAuth access token from the docs UI, use `POST /institutions/oauth/token` with the JSON body already supplied in the reference:

```json
{
  "grant_type": "client_credentials",
  "institutionId": "{{institutionId}}",
  "client_assertion_type": "urn:ietf:params:oauth:client-assertion-type:jwt-bearer",
  "client_assertion": "paste-client-assertion-jwt-here"
}
```

## 5. PIN And Encryption Flows

Card creation, PIN change, and PIN reset all expect encrypted payload fragments. A `Generate RSA Key` flow is also expected for encryption-related operations, but its route still needs to be confirmed before it can be exposed as a callable reference page.
